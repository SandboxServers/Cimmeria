# Data-Driven Content Engine

> **Last updated**: 2026-03-02
> **Sources**: All 20 mission scripts, 5 effect scripts, 10 space scripts, `db/resources.sql` schema, `python/common/Event.py`, `python/cell/AbilityManager.py`, `python/cell/SGWMob.py`, `python/cell/Lootable.py`

---

## Executive Summary

Cimmeria has **1,040 missions**, **3,216 effects**, **5,411 dialogs**, and **6,059 items** defined in the database. Of these, only **16 missions** (1.5%) and **4 effects** (0.12%) have the Python scripts they need to function. Every one of those scripts follows the same mechanical pattern: subscribe to an event, check a condition, execute an action. There are no algorithms, no complex state machines, no creative logic — just wiring.

This document designs a **data-driven content engine** that replaces per-script Python files with database rows interpreted by a generic runtime. The result: content creators define game behavior by inserting trigger/condition/action rows into PostgreSQL, and the server knows kung fu.

The engine covers six systems: **missions**, **effects**, **dialogs**, **spawns**, **loot**, and **space events**. Together these account for 100% of the auto-generated Python scripts and ~60% of the hand-written space scripts.

## The Problem in Detail

### What Scripts Actually Do

Every auto-generated mission script in the codebase follows this exact pattern:

```python
# Generated by Atrea Script Editor
class ArmYourself(Script):
    def __init__(self, owner):
        # Step 1: Subscribe to events
        self.owner.subscribe("entity.interact.tag::Cellblock_WoodenCrate", self.onInteract)
        self.owner.subscribe("dialog.open::3995", self.onDialog)

    def onInteract(self, args):
        # Step 2: Check a condition
        status = self.owner.missions.getStepStatus(622, 2114)
        if status == Constants.MISSION_Active:
            # Step 3: Execute an action
            self.owner.missions.advance(622, 2114)
            self.owner.inventory.addItem(3, 55, 1)
            self.owner.displayDialog(None, 3995)
```

That's it. Subscribe → Check → Act. Every mission script. Every effect script. Every space script trigger. The Atrea Script Editor generated hundreds of these, and each one became a separate Python file that must exist on disk for the mission to function.

## Engine Architecture

### Overview

```
┌─────────────────────────────────────────────────────────┐
│                    PostgreSQL                            │
│                                                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ triggers │  │conditions│  │ actions  │  (new tables) │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘              │
│       │              │             │                    │
│  ┌────┴──────────────┴─────────────┴────┐               │
│  │          trigger_chains              │  (new table)  │
│  └──────────────────────────────────────┘               │
│                                                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ missions │  │ dialogs  │  │ effects  │  (existing)   │
│  └──────────┘  └──────────┘  └──────────┘              │
└────────────────────┬────────────────────────────────────┘
                     │
                     │  Loaded at startup / hot-reloadable
                     ▼
┌─────────────────────────────────────────────────────────┐
│              Content Engine Runtime                      │
│                                                         │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────────┐ │
│  │ Event Router │  │  Condition   │  │    Action     │ │
│  │              │→ │  Evaluator   │→ │   Executor    │ │
│  │ (subscribe/  │  │              │  │               │ │
│  │  dispatch)   │  │ (check state)│  │ (modify world)│ │
│  └──────────────┘  └──────────────┘  └───────────────┘ │
│                                                         │
│  ┌──────────────────────────────────────────────────┐   │
│  │              Trigger Context                      │   │
│  │  player, entity, space, mission_state, counters  │   │
│  └──────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
         │                    │                  │
         ▼                    ▼                  ▼
   Event System         Game Systems        Client RPC
   (subscribe/fire)     (missions,          (displayDialog,
                         inventory,          playSequence,
                         combat)             onSystemMsg)
```

### Data Flow

1. **Startup**: Load all trigger chains from DB into memory, indexed by event type
2. **Player enters zone**: Register zone-scoped triggers for that player
3. **Event fires** (player interacts with NPC, enters region, etc.):
   a. Event Router looks up matching trigger chains by event key
   b. Condition Evaluator checks each chain's conditions against current state
   c. Action Executor runs the actions for chains that pass conditions
4. **Actions modify state**: Advance missions, grant items, show dialogs, etc.
5. **State changes fire more events**: Mission completion triggers next chain

## Database Schema

### Core Tables

#### `content_triggers` — What events to listen for

```sql
CREATE TABLE content_triggers (
    trigger_id    serial PRIMARY KEY,
    chain_id      integer NOT NULL,           -- Groups triggers into logical chains
    event_type    varchar(50) NOT NULL,        -- 'interact_tag', 'enter_region', 'dialog_choice', etc.
    event_key     varchar(200) NOT NULL,       -- Tag name, region name, dialog ID, item ID, etc.
    scope         varchar(20) NOT NULL         -- 'player', 'space', 'global'
        DEFAULT 'player',
    once          boolean NOT NULL             -- Unsubscribe after first fire?
        DEFAULT false,
    sort_order    integer NOT NULL DEFAULT 0   -- Ordering within a chain
);

CREATE INDEX idx_triggers_event ON content_triggers (event_type, event_key);
CREATE INDEX idx_triggers_chain ON content_triggers (chain_id);
```

#### `content_conditions` — What state to check before executing

```sql
CREATE TABLE content_conditions (
    condition_id  serial PRIMARY KEY,
    chain_id      integer NOT NULL,
    condition_type varchar(50) NOT NULL,       -- 'mission_status', 'step_status', 'has_item', etc.
    target_id     integer,                     -- Mission ID, item ID, etc.
    target_key    varchar(200),                -- Secondary key (step ID, objective ID)
    operator      varchar(10) NOT NULL         -- 'eq', 'neq', 'gte', 'lte', 'contains'
        DEFAULT 'eq',
    value         varchar(200) NOT NULL,       -- Expected value ('active', 'completed', 'not_active')
    sort_order    integer NOT NULL DEFAULT 0
);

CREATE INDEX idx_conditions_chain ON content_conditions (chain_id);
```

#### `content_actions` — What to do when conditions pass

```sql
CREATE TABLE content_actions (
    action_id     serial PRIMARY KEY,
    chain_id      integer NOT NULL,
    action_type   varchar(50) NOT NULL,        -- 'accept_mission', 'advance_step', 'add_item', etc.
    target_id     integer,                     -- Mission ID, item ID, dialog ID, etc.
    target_key    varchar(200),                -- Secondary key (step ID, container ID)
    params        jsonb DEFAULT '{}',          -- Additional parameters
    delay_ms      integer NOT NULL DEFAULT 0,  -- Delay before execution (for sequencing)
    sort_order    integer NOT NULL DEFAULT 0
);

CREATE INDEX idx_actions_chain ON content_actions (chain_id);
```

#### `content_chains` — Metadata grouping triggers, conditions, and actions

```sql
CREATE TABLE content_chains (
    chain_id      serial PRIMARY KEY,
    name          varchar(200),                -- Human-readable description
    scope_type    varchar(50) NOT NULL,        -- 'mission', 'space', 'effect', 'global'
    scope_id      integer,                     -- Mission ID, space/world ID, effect ID
    enabled       boolean NOT NULL DEFAULT true,
    priority      integer NOT NULL DEFAULT 0,  -- Higher = evaluated first
    description   text                         -- Author notes
);

CREATE INDEX idx_chains_scope ON content_chains (scope_type, scope_id);
```

#### `content_counters` — Per-player counters for multi-step tracking (kill X of Y)

```sql
CREATE TABLE content_counters (
    counter_id    serial PRIMARY KEY,
    chain_id      integer NOT NULL,
    counter_name  varchar(100) NOT NULL,       -- Unique within chain
    target_value  integer NOT NULL,            -- Trigger when counter reaches this
    reset_on      varchar(50)                  -- 'mission_complete', 'zone_change', 'never'
        DEFAULT 'mission_complete'
);

CREATE INDEX idx_counters_chain ON content_counters (chain_id);
```

### Relationship Diagram

```
content_chains (1) ──── (N) content_triggers     "When this event happens..."
       │
       ├──────────────── (N) content_conditions   "...and these are true..."
       │
       ├──────────────── (N) content_actions       "...do these things."
       │
       └──────────────── (N) content_counters      "Track progress toward a goal."


content_chains.scope_type + scope_id links to:
  'mission'  → missions.mission_id          (chain belongs to a mission)
  'space'    → worlds.world_id              (chain belongs to a zone)
  'effect'   → effects.effect_id            (chain belongs to an effect)
  'global'   → NULL                         (always active)
```

## Condition Type Reference

| condition_type | target_id | target_key | operator | value | Meaning |
|---|---|---|---|---|---|
| `mission_status` | Mission ID | — | eq/neq | `active`/`completed`/`not_active` | Mission state check |

| `archetype` | — | — | eq/neq/lt/gt | Archetype ID | Player class check |
| `counter` | — | Counter name | gte/eq | Target count | Counter threshold |
| `entering` | — | — | eq | `true`/`false` | Region enter vs exit |
| `level` | — | — | gte/lte | Level number | Player level check |
| `always` | — | — | — | — | No condition (always passes) |

### Condition Logic

Multiple conditions on the same chain use **AND** logic — all must pass. For **OR** logic, create multiple chains with different conditions but the same actions.

## Effect System Data Model

Effects have a simpler pattern than missions. Every effect script does one of three things on pulse:

1. **Deal damage**: Call `qrCombatDamage(stat, school, baseDmg, crit, resist)`, apply result
2. **Modify stats**: Call `changeStat(statId, min, max)`
3. **Apply/remove other effects**: Call `addEffect(id)` or `removeEffect(id)`

The effect parameters are already stored in the `effects` table's `tcm_param1`/`tcm_param2` columns and the Atrea node graph. The data-driven model makes this explicit:

```sql
-- Example: RangedPhysicalDamage effect (currently 60 lines of Python)
INSERT INTO content_chains VALUES (100, 'Ranged Physical Damage', 'effect', 1234);

INSERT INTO content_triggers VALUES (DEFAULT, 100, 'effect_pulse_begin', '1234', 'player', false, 0);

INSERT INTO content_actions VALUES
(DEFAULT, 100, 'qr_combat_damage', 15, NULL,
 '{"school": 3, "base_param": "HealthDamage", "use_crit": true, "use_resist": true}', 0);
```

One chain, one trigger, one action. That's it. The `qr_combat_damage` action executor reads the `base_param` from the effect's parameter dictionary (already populated in the DB), runs the QR calculation (which is already in C++ via `AbilityManager`), and applies the result.

For the 3,216 effects in the database, the vast majority are simple damage or stat modification effects that map to 1-3 action rows each. The 4 existing effect scripts total 845 lines of Python. Their data-driven equivalents total ~12 rows.

## Spawn System Data Model

The `spawn_sets` and `spawn_points` tables exist in the schema but have **zero rows**. Currently, NPCs are spawned by space scripts via `createEntity(template, position)`. The data-driven model makes spawn definitions declarative:

```sql
-- Extend existing (empty) spawn tables rather than creating new ones
-- These tables already exist with the right structure

-- Populate spawn_sets: which NPCs can spawn in which zones
INSERT INTO spawn_sets (set_id, name, type, world_id, height, radius) VALUES
(1, 'CellBlock Guards',      'hostile',  1, 2.0, 5.0),
(2, 'CellBlock Interactable', 'static',  1, 0.0, 0.0),
(3, 'CellBlock Quest NPCs',  'friendly', 1, 0.0, 0.0);

-- Populate spawn_points: exact positions for each NPC
INSERT INTO spawn_points (point_id, x, y, z, orientation, spawn_set_name, spawn_table_name) VALUES
(1, 150.3, 22.0, -84.5, 180.0, 'CellBlock Guards', 'template_4'),   -- PRU guard
(2, 142.7, 22.0, -91.2, 90.0,  'CellBlock Guards', 'template_4'),   -- PRU guard
(3, 155.0, 22.0, -80.0, 0.0,   'CellBlock Interactable', 'template_13'); -- Wooden crate

-- New: spawn_table linking spawn names to entity templates with weights
CREATE TABLE spawn_tables (
    table_id      serial PRIMARY KEY,
    table_name    varchar(100) NOT NULL,
    template_id   integer NOT NULL REFERENCES entity_templates(template_id),
    weight        real NOT NULL DEFAULT 1.0,
    min_level     integer,
    max_level     integer,
    tag           varchar(200),               -- Entity tag for trigger system
    respawn_ms    integer DEFAULT 300000       -- 5 minute default respawn
);
```

Space scripts currently contain hardcoded spawn coordinates for every NPC. Those coordinates can be extracted directly from the scripts (positions are literal float values in every `createEntity` call) and inserted into `spawn_points`. The engine's spawn manager reads these tables at zone startup and creates entities — no Python required.

## Dialog Gating Data Model

The `dialog_set_maps` table already has the columns needed for mission-based dialog gating:

```sql
-- Already exists:
missions_completed integer[] DEFAULT '{}'::integer[]
missions_not_accepted integer[] DEFAULT '{}'::integer[]
alignments integer[] DEFAULT '{}'::integer[]
factions integer[] DEFAULT '{}'::integer[]
```

These columns are **100% empty** across all 1,178 dialog set maps. The system was designed for data-driven dialog gating but never populated. Filling these arrays enables conditional dialog display without any code:

```sql
-- Show "Search for Frost" dialog only after mission 622 is completed
UPDATE dialog_set_maps
SET missions_completed = '{622}'
WHERE dialog_set_map_id = 1001;

-- Hide "Arm Yourself" dialog once mission 622 is accepted
UPDATE dialog_set_maps
SET missions_not_accepted = '{622}'
WHERE dialog_set_map_id = 1002;
```

The dialog system code in the Python layer (or its replacement) checks these arrays when building the dialog option list for a player. If `missions_completed` is non-empty, only show the option to players who have completed all listed missions. If `missions_not_accepted` is non-empty, hide the option from players who have started any listed mission.

**This is the single lowest-effort, highest-impact change in the entire project.** Populating ~500 rows of `missions_completed` and `missions_not_accepted` arrays would enable mission-gated dialog across every NPC that has dialog sets — covering a substantial portion of the 5,411 dialogs.

## Loot Table Generation

The loot system has the correct code (`Lootable.py`, 221 lines) but only **3 entries in 2 tables**. The data needed to generate proper loot tables already exists:

- **6,059 items** with levels, types, rarity flags, and archetype requirements
- **153 entity templates** with levels and factions
- **Level-based zones** with known progression (L1 tutorial → L50 endgame)

Loot table generation is algorithmic:

```sql
-- Generate loot table for a level-5 hostile NPC
INSERT INTO loot_tables (loot_table_id, description) VALUES
(100, 'Level 5 Hostile NPC - Common Drops');

-- Items within ±3 levels of the NPC, weighted by rarity
INSERT INTO loot (loot_table_id, design_id, min_quantity, probability, max_quantity)
SELECT
    100,
    i.item_id,
    1,
    CASE
        WHEN i.flags & 1 = 1 THEN 0.01   -- Rare: 1%
        WHEN i.flags & 2 = 2 THEN 0.05   -- Uncommon: 5%
        ELSE 0.15                          -- Common: 15%
    END,
    1
FROM items i
WHERE i.level BETWEEN 2 AND 8             -- NPC level ± 3
  AND i.item_id NOT IN (                   -- Exclude quest items
      SELECT item_id FROM mission_rewards
  );

-- Currency drop (design_id = NULL means naquadah)
INSERT INTO loot (loot_table_id, design_id, min_quantity, probability, max_quantity) VALUES
(100, NULL, 5, 0.80, 15);                 -- 80% chance of 5-15 naquadah
```

Assigning loot tables to entity templates:

```sql
-- All hostile NPCs level 1-10 get the same basic loot table
UPDATE entity_templates
SET loot_table_id = 100
WHERE class = 'SGWMob'
  AND level BETWEEN 1 AND 10
  AND faction IN (10, 11);                 -- Hostile factions
```

## Runtime Implementation

The runtime interpreter can be implemented in the current Python scripting layer without any C++ changes. Here is the architecture.

### ContentEngine Class

```python
# python/cell/ContentEngine.py
class ContentEngine:
    """
    Data-driven content interpreter. Replaces per-mission/effect Python scripts
    with a generic trigger→condition→action pipeline driven by DB rows.
    """

    def __init__(self):
        self.chains = {}          # chain_id → ChainDef
        self.event_index = {}     # (event_type, event_key) → [chain_id, ...]
        self.player_state = {}    # entity_id → {counters, active_subscriptions}

    def load(self):
        """Load all chains from database at startup."""
        rows = Atrea.dbQuery("SELECT * FROM content_chains WHERE enabled = true")
        for row in rows:
            chain = ChainDef(row)
            chain.triggers = self._load_triggers(chain.chain_id)
            chain.conditions = self._load_conditions(chain.chain_id)
            chain.actions = self._load_actions(chain.chain_id)
            chain.counters = self._load_counters(chain.chain_id)
            self.chains[chain.chain_id] = chain

            for trigger in chain.triggers:
                key = (trigger.event_type, trigger.event_key)
                self.event_index.setdefault(key, []).append(chain.chain_id)

    def register_player(self, player, world_name):
        """Register a player for content triggers in their current zone."""
        state = PlayerContentState(player)
        self.player_state[player.entityId] = state

        # Subscribe to all events relevant to this player's zone and missions
        for chain_id, chain in self.chains.items():
            if self._chain_applies(chain, player, world_name):
                for trigger in chain.triggers:
                    event_str = self._make_event_string(trigger)
                    sub_id = player.subscribe(event_str,
                        lambda args, c=chain, t=trigger: self._on_event(player, c, t, args),
                        once=trigger.once)
                    state.subscriptions.append(sub_id)

    def _on_event(self, player, chain, trigger, args):
        """Event fired — evaluate conditions and execute actions."""
        if not self._evaluate_conditions(chain.conditions, player, args):
            return False

        for action in sorted(chain.actions, key=lambda a: a.sort_order):
            if action.delay_ms > 0:
                Atrea.addTimer(action.delay_ms / 1000.0,
                    lambda: self._execute_action(action, player, args))
            else:
                self._execute_action(action, player, args)

        return True

    def _evaluate_conditions(self, conditions, player, args):
        """All conditions must pass (AND logic)."""
        for cond in conditions:
            if not self._check_condition(cond, player, args):
                return False
        return True

    def _check_condition(self, cond, player, args):
        """Evaluate a single condition against player state."""
        if cond.condition_type == 'mission_status':
            status = player.missions.getMissionStatus(cond.target_id)
            return self._compare(status, cond.operator, cond.value)

        elif cond.condition_type == 'step_status':
            status = player.missions.getStepStatus(cond.target_id, int(cond.target_key))
            return self._compare(status, cond.operator, cond.value)

        elif cond.condition_type == 'objective_status':
            status = player.missions.getObjectiveStatus(cond.target_id, int(cond.target_key))
            return self._compare(status, cond.operator, cond.value)

        elif cond.condition_type == 'has_item':
            item = player.inventory.findItem(cond.target_id)
            has = item is not None
            return self._compare(has, cond.operator, cond.value == 'true')

        elif cond.condition_type == 'archetype':
            return self._compare(player.archetype, cond.operator, int(cond.value))

        elif cond.condition_type == 'entering':
            return self._compare(args.get('entering', False), cond.operator, cond.value == 'true')

        elif cond.condition_type == 'counter':
            state = self.player_state.get(player.entityId)
            count = state.counters.get(cond.target_key, 0) if state else 0
            return self._compare(count, cond.operator, int(cond.value))

        elif cond.condition_type == 'always':
            return True

        return False

    def _execute_action(self, action, player, args):
        """Execute a single action."""
        if action.action_type == 'accept_mission':
            player.missions.accept(action.target_id)

        elif action.action_type == 'advance_step':
            player.missions.advance(action.target_id, int(action.target_key))

        elif action.action_type == 'complete_mission':
            player.missions.complete(action.target_id)

        elif action.action_type == 'complete_objective':
            player.missions.completeObjective(action.target_id, int(action.target_key))

        elif action.action_type == 'fail_mission':
            player.missions.fail(action.target_id)

        elif action.action_type == 'add_item':
            qty = action.params.get('quantity', 1)
            container = int(action.target_key) if action.target_key else 0
            player.inventory.addItem(container, action.target_id, qty)
            player.inventory.flushUpdates()

        elif action.action_type == 'remove_item':
            qty = action.params.get('quantity', 1)
            player.inventory.removeItemByDesign(action.target_id, qty, False)
            player.inventory.flushUpdates()

        elif action.action_type == 'display_dialog':
            npc = None
            npc_tag = action.params.get('npc_tag')
            if npc_tag:
                npc = self._find_entity_by_tag(player, npc_tag)
            player.displayDialog(npc, action.target_id)

        elif action.action_type == 'add_dialog_option':
            dialog_set = DefMgr.get('interaction_set_map', action.target_id)
            if dialog_set:
                mission_id = action.params.get('mission_id')
                player.addDialog(int(action.target_key), dialog_set, mission_id)

        elif action.action_type == 'remove_dialog_option':
            dialog_set = DefMgr.get('interaction_set_map', action.target_id)
            if dialog_set:
                player.removeDialog(int(action.target_key), dialog_set)

        elif action.action_type == 'play_sequence':
            target = None
            target_tag = action.params.get('target_tag')
            if target_tag:
                target = self._find_entity_by_tag(player, target_tag)
            target_id = target.entityId if target else None
            player.playSequence(action.target_id, target_id, True, 0, 0, 0, True, True)

        elif action.action_type == 'set_interaction_type':
            entity = self._find_entity_by_tag(player, action.target_key)
            if entity:
                flags = action.params.get('flags', 0)
                op = action.params.get('op', 'or')
                if op == 'or':
                    entity.setInteractionType(entity.interactionType | flags)
                elif op == 'and_not':
                    entity.setInteractionType(entity.interactionType & ~flags)

        elif action.action_type == 'set_aggression':
            entity = self._find_entity_by_tag(player, action.target_key)
            if entity:
                entity.setAggression(action.params.get('level', 0))

        elif action.action_type == 'set_visible':
            entity = self._find_entity_by_tag(player, action.target_key)
            if entity:
                entity.setVisible(action.params.get('visible', True))

        elif action.action_type == 'move_entity':
            entity = self._find_entity_by_tag(player, action.target_key)
            if entity:
                pos = Atrea.Vector3(action.params['x'], action.params['y'], action.params['z'])
                entity.addWaypoint(pos, lambda: None)

        elif action.action_type == 'spawn_entity':
            tpl = DefMgr.get('template', action.target_id)
            if tpl:
                pos = Atrea.Vector3(action.params['x'], action.params['y'], action.params['z'])
                entity = player.space.createEntity(tpl, pos)
                tag = action.params.get('tag')
                if tag and entity:
                    entity.tag = tag

        elif action.action_type == 'destroy_entity':
            entity = self._find_entity_by_tag(player, action.target_key)
            if entity:
                entity.destroy()

        elif action.action_type == 'system_message':
            msg_type = action.params.get('type', 11)
            player.client.onSystemCommunication(msg_type, action.target_id, '', [])

        elif action.action_type == 'start_minigame':
            name = action.params.get('name', 'Livewire')
            difficulty = action.params.get('difficulty', 1)
            game_ids = [k for k, v in Config.MINIGAME_NAMES.items() if v == name]
            if game_ids:
                mg = Minigame('', difficulty, game_ids[0], 1, 0xffff,
                    lambda mg, rc: self._on_minigame_result(player, chain, rc))
                player.requestStartMinigame(mg)

        elif action.action_type == 'increment_counter':
            state = self.player_state.get(player.entityId)
            if state:
                name = action.target_key
                amount = action.params.get('amount', 1)
                state.counters[name] = state.counters.get(name, 0) + amount
                # Check if any counter thresholds are met
                self._check_counter_thresholds(player, name, state.counters[name])

        elif action.action_type == 'reset_counter':
            state = self.player_state.get(player.entityId)
            if state:
                state.counters[action.target_key] = 0

    def _compare(self, actual, operator, expected):
        """Generic comparison operator."""
        STATUS_MAP = {'active': 0, 'completed': 1, 'not_active': 2}
        if isinstance(expected, str) and expected in STATUS_MAP:
            expected = STATUS_MAP[expected]

        if operator == 'eq':  return actual == expected
        if operator == 'neq': return actual != expected
        if operator == 'gt':  return actual > expected
        if operator == 'gte': return actual >= expected
        if operator == 'lt':  return actual < expected
        if operator == 'lte': return actual <= expected
        return False
```

This is ~200 lines of Python. It replaces ~4,600 lines of auto-generated scripts and ~6,000 lines of space scripts. It runs within the existing Cimmeria architecture with zero C++ changes.

### Integration Points

The engine hooks into two existing systems:

1. **Space scripts** (`python/cell/spaces/*.py`): Replace the mission/dialog/spawn logic with `ContentEngine.register_player()` calls. Space scripts still handle zone-specific setup that isn't mission-related.

2. **Effect manager** (`python/cell/EffectManager.py`): When an effect has no `script_name` but has content chains with `scope_type='effect'`, the engine handles pulse/init/removal events.

### Hot Reload

Because chains are loaded from the database, content changes are hot-reloadable:

```python
# Admin command to reload content without server restart
def reloadContent():
    engine.chains.clear()
    engine.event_index.clear()
    engine.load()
    # Re-register all online players
    for player in Atrea.findAllEntities():
        if hasattr(player, 'space'):
            engine.register_player(player, player.space.worldName)
```

## Migration Path

### Step 1: Build the Engine (~1 week)

Implement `ContentEngine.py` (~200 lines) and the 5 SQL tables. Deploy alongside existing scripts — the engine and scripts can coexist because they subscribe to the same events independently.

### Step 2: Migrate Castle_CellBlock (~1 week)

Convert all 15 Castle_CellBlock Python scripts to database rows. This is the most complete tutorial chain and serves as the validation test. Run both old scripts and new engine side by side, verify identical behavior, then disable the old scripts.

Expected: ~45 chains, ~250 rows replacing ~2,100 lines of Python.

### Step 3: Migrate SGC_W1 (~3 days)

Convert SGC_W1 scripts. Smaller scope (3 missions), validates the engine with a different zone.

Expected: ~15 chains, ~80 rows replacing ~800 lines of Python.

### Step 4: Populate Unscripted Missions (~2-4 weeks)

This is where the engine pays for itself. The database has **1,024 unscripted missions** with full step/objective/task structure and rich descriptive text. For each:

1. Parse step text to identify trigger type ("Speak to X" → `interact_tag`, "Go to Y" → `enter_region`, "Kill X" → `entity_dead_tag`)
2. Create chains with appropriate conditions (step/objective status checks)
3. Create actions (advance step, complete objective, display dialog)
4. Cross-reference entity templates and speakers for NPC tags

The [mission-chains.md](../content/mission-chains.md) document identified ~300 missions as SCRIPTABLE — they have rich enough step text to generate chains programmatically. An LLM can accelerate this by parsing step text and generating SQL INSERT statements.

### Step 5: Populate Effects (~1 week)

For each of the 3,216 effects:

1. Read the effect's `tcm_param1`/`tcm_param2` for damage/stat parameters
2. Read the effect's `pulse_count`/`pulse_duration` for timing
3. Generate 1-3 action rows per effect (damage, stat change, apply/remove)

Most effects are simple damage or stat modifications. The 4 existing effect scripts provide templates for the ~10 distinct effect behavior patterns.

### Step 6: Populate Spawns (~1 week)

1. Extract NPC positions from the 10 existing space scripts (literal coordinates in `createEntity` calls)
2. Insert into `spawn_points` and `spawn_sets`
3. For zones without space scripts, use client map data to infer NPC placement regions
4. Generate loot tables from item/mob level cross-references

### Step 7: Populate Dialog Gating (~3 days)

1. For each mission with dialog references, populate `dialog_set_maps.missions_completed` and `missions_not_accepted` arrays
2. Cross-reference mission chains to determine which dialogs should be gated on which missions
3. This enables NPC dialogs to change based on player progress — currently impossible because all arrays are empty

### Step 8: LLM-Assisted Content Generation (Ongoing)

With the engine running, use LLMs to generate content at scale:

- Feed mission step text + entity template names + zone data to an LLM
- LLM generates SQL INSERT statements for trigger chains
- Human reviews and adjusts
- Insert into database, hot-reload, test in-game

This turns the ~300 SCRIPTABLE missions from "needs a programmer" into "needs a reviewer."

## What the Engine Does NOT Replace
